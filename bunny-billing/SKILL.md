# Bunny Billing

Invoices, payments, and credit notes are Bunny's record of who owes
what and who paid. The subscription cron produces most of this data
automatically; this skill covers the state machines, the manual
mutations, and the patterns for corrections, refunds, and reconciliation.

Public docs: [docs.bunny.com/developer](https://docs.bunny.com/developer).

## Credential safety

Billing mutations move real money. A stray `invoiceRefund` issues a
refund through your payment gateway within seconds.

- **Load tokens from environment variables only.** Never hardcode, never
  log, never commit.
- **Scope tokens to `billing:write`** for anything that creates or
  modifies invoices / payments / credits. `standard:read` is enough for
  dashboards.
- **Keep a two-person rule** in production code: refund, void, and
  credit-issue flows should require a second internal approval (your
  app's audit trail — Bunny doesn't gate these natively).
- **Rotate tokens** on a cadence and immediately on suspected leak.

## Invoice state machine

Invoices pass through a 12-state machine. Most happy-path flows touch
only `PREPARING → READY → NOT_DUE → DUE → PAID`.

```text
            subscription apply / cron tick
                         │
                         ▼
                   ┌──────────┐
                   │PREPARING │   tax calc, line-item build
                   └────┬─────┘
                        │ set_ready
                        ▼
                   ┌──────────┐
                   │  READY   │   Bunny has the invoice, not issued yet
                   └────┬─────┘
                        │ issue
                        ▼
                   ┌──────────┐
                   │ NOT_DUE  │   delivered, due date in the future
                   └────┬─────┘
                        │ (due date arrives)
                        ▼
                   ┌──────────┐  charge-on-file runs  ┌────────────────────┐
                   │   DUE    │ ────────────────────► │ PROCESSING_PAYMENT │
                   └────┬─────┘                       └──────┬─────────────┘
                        │ (payment received)                 │
                        ▼                                    ▼
                   ┌──────────┐                        ┌──────────┐
                   │  UNPAID  │                        │   PAID   │  (terminal)
                   └──────────┘                        └──────────┘

Side branches:
  DRAFT      → edit line items before PREPARING
  FAILED     → tax plugin couldn't calc; invoiceRetryTax or invoiceUpdateTaxes
  VOIDED     → invoiceVoid (non-recoverable)
  MERGED     → combined into another invoice via invoiceMerge (reversible)
  QUEUED     → waiting for consolidated billing run
  CONSOLIDATED → joined a consolidated invoice (terminal)
```

State transitions are driven by **state-machine events** on the server;
most mutations wrap one of them (`invoiceIssue` fires the `issue`
event, `invoiceVoid` fires `void`, etc.). You can't arbitrarily move
between states — the mutation will fail if the event isn't valid from
the current state.

## How invoices get created

In order of how often you'll see each path:

### 1. Automatically on subscription apply

`quoteApplyChanges` / `subscriptionCreate` produce the first invoice
synchronously; see the `invoice { id number amount }` payload in
`bunny-quoting`. The subscription cron then produces subsequent
invoices at each billing period boundary.

### 2. On-demand for the next period

When you need to bill ahead of the cron (demo, mid-period catch-up,
usage-heavy period):

```graphql
mutation GenerateNext {
  invoiceGenerateNextPeriod(
    accountId: "acc_xxx"
    billUsageCharges: true             # include unbilled usage
  ) {
    invoice { id number state amount }
    errors
  }
}
```

Preconditions (both enforced server-side; skip either and the mutation
errors clearly):

- **Account must allow draft invoices** — the mutation errors with
  `Account draft invoices must be enabled` when the feature isn't set
  on the account. Configure via the admin portal or
  `accountUpdate(id, attributes: { draftInvoicesEnabled: true })`
  before calling.
- **`billUsageCharges: true`** requires usage charges to be enabled at
  the warren level. If disabled, the mutation errors with `Usage
  charges are not supported yet`; fall back to `false` and let the
  cron pick up usage on its normal schedule.

### 3. Manually by editing a DRAFT

Draft invoices are rare (ops tooling, one-off charges). Build
line-items via `invoiceItemCreate` / `invoiceItemUpdate` under a DRAFT
invoice, then call `invoiceRegenerate` to move it into the normal
flow.

## The corrections trilemma: void vs credit vs refund

These three look similar but do different things. Pick based on the
state of the invoice and what you want visible on the customer's
account:

| You want to … | Invoice state | Use | Effect |
| --- | --- | --- | --- |
| Make an unpaid invoice go away (mistake, fixed pricing before billing) | `NOT_DUE` / `DUE` / `UNPAID` | `invoiceVoid(id)` | Invoice → `VOIDED`; no credit, no refund. Customer owes nothing. |
| Credit a paid invoice because the customer was overcharged | `PAID` | `invoiceCredit(invoiceId, disputeReasonId?)` | Creates a CreditNote for the full amount; funds available as credit on the account. No gateway refund. |
| Refund a paid invoice back to the customer's card | `PAID` | `invoiceRefund(invoiceId, disputeReasonId?)` | Refunds via the payment gateway + creates a CreditNote. Funds leave your balance. Payload returns `{ creditNote, errors }` — **no `invoice` field**; re-query the invoice if you need its new state. |
| Credit a partial amount as account balance | `PAID` | `invoiceAddCredit(id, amount)` | Adds a credit line; doesn't produce a full CreditNote. |

**Void is one-way** — there's no `invoiceUnvoid`. If you need the
invoice back, use `invoiceRegenerate(id)` on the originating
subscription's period.

**Refund makes real money move.** Your payment plugin (Stripe,
Adyen, etc.) processes the refund; a gateway failure surfaces in
`invoice.paymentFailureReason`.

## Credit notes

A CreditNote is Bunny's formal record of credit issued against a paid
invoice — used to reconcile against accounting, support refund
workflows, and let customers redeem the balance against future invoices.

```text
                  invoiceCredit / invoiceRefund
                         │
                         ▼
                    ┌─────────┐
                    │  DRAFT  │   build/adjust credit-note items
                    └────┬────┘
                         │ creditNoteIssue
                         ▼
                    ┌─────────┐
                    │  READY  │   available to apply
                    └────┬────┘
                         │ creditNoteApplyToInvoices
                         ▼
     ┌───────────────────────────────────────────┐
     │ PARTIALLY_APPLIED  ──────────────►  APPLIED│  (balance exhausted)
     └───────────────────────────────────────────┘
```

Explicit creation (when you want control over line items):

```graphql
mutation CreateCreditNote {
  creditNoteCreate(
    invoiceId: "inv_xxx"
    kind: WRITE_OFF                    # CREDIT | REFUND | WRITE_OFF; optional
    disputeReasonId: "dsp_xxx"         # optional — tags the reason
  ) {
    creditNote { id state kind }
    errors
  }
}

mutation IssueCreditNote {
  # Note: CreditNoteIssuePayload has no `errors` field — don't select it
  creditNoteIssue(id: "crn_xxx") {
    creditNote { id state number }
  }
}

mutation ApplyCredit {
  creditNoteApplyToInvoices(
    creditNoteId: "crn_xxx"
    invoiceIds: ["inv_abc", "inv_def"]   # one or many; applied in order
  ) {
    creditNote { id state amountUnapplied }
    errors
  }
}
```

`CreditNoteKind` (optional — defaults when omitted):

- **`WRITE_OFF`** — the common case; writes off an invoice without
  moving money through the gateway. Use for corrections, goodwill
  credits, and any "make this invoice go away" scenario on a paid or
  unpaid balance. Cannot be *applied* to other invoices via
  `creditNoteApplyToInvoices` — it's consumed by the originating
  invoice only.
- **`REFUND`** — creates a credit note tied to a gateway refund.
  Only accepted on invoices that have received payments (the
  mutation rejects otherwise with `Invoice must have received
  payments to create a refund credit note`). Typically produced
  automatically by `invoiceRefund`; rarely constructed manually.
- **`CREDIT`** — reserved for account-level credit that draws down
  against future invoices. Server returns `Kind credit is not
  supported yet` in the current release; treat as unavailable until
  Bunny ships it.

Behavioural choices — full write-off vs. partial line-item credit vs.
post-issue credit — are **lifecycle choices** (stop at DRAFT and edit
line items for partial; go through `creditNoteIssue` for full), not
`kind` values. Don't conflate them.

`creditNoteIssue(id)` takes only the id — no `emailContactIds`, no
other arguments.

`creditNoteApplyToInvoices` is meaningful only for non-`WRITE_OFF`
kinds. `WRITE_OFF` credit notes cannot be redistributed across
invoices — the apply-call rejects with `Credit note cannot be applied
to invoices with write-off credit notes`. The "apply against any
unpaid invoice on the same account" pattern is specifically for
`REFUND` (and `CREDIT`, once supported).

## Payments

`Payment` is money received; `PaymentApplication` is the join between
a payment and an invoice. One payment can span multiple invoices.

### Apply a single payment to one invoice

```graphql
mutation ApplyToOne {
  paymentApplyToInvoice(paymentId: "pay_xxx", invoiceId: "inv_xxx") {
    invoice { id state }
    payment { id amount }
    errors
  }
}
```

`PaymentApplyToInvoicePayload` returns `{ invoice, payment, errors }`
— there's no `paymentApplication` field on this single-invoice
variant. Query the invoice or payment to confirm the new state.

### Apply to multiple invoices (lump-sum payment)

Common when a customer pays for two overdue months in one transaction:

```graphql
mutation ApplyToMany {
  paymentApplyToInvoices(
    paymentId: "pay_xxx"
    invoiceIds: ["inv_may", "inv_june"]
  ) {
    payment { id amount }
    paymentApplications { id amount }
    paidInvoices { id number state }
    removedInvoices { id }
    errors
  }
}
```

Bunny applies the payment to invoices in the order given, up to each
invoice's balance, until the payment amount is exhausted. Remainder
stays on the payment as unapplied. `PaymentApplicationType` exposes
`{ id, amount, paymentId, invoiceId? }` among other fields — the
nested `paymentApplications` selection here is intentionally minimal
because the `paidInvoices` / `removedInvoices` arrays tell you most of
what you want to know post-application.

### Hold money as account credit without an invoice

```graphql
mutation ApplyToAccount {
  paymentApplyToAccount(paymentId: "pay_xxx", accountId: "acc_xxx") {
    payment { id amount }
    errors
  }
}
```

`PaymentApplyToAccountPayload` returns `{ payment, errors }` only
— like the single-invoice variant, no `paymentApplication` on this
payload.

Useful for prepayments, deposits, or customer overpayments you want
to absorb and later apply to future invoices.

### Reverse an application

`paymentUnapplyFromInvoices(paymentId, invoiceIds)` moves the
application off those invoices — use when you applied a payment to
the wrong invoice and need to correct without deleting.

### Payment methods

```graphql
mutation AddCard {
  paymentMethodCreate(
    accountId: "acc_xxx"
    pluginGuid: "plg_stripe_xxx"       # which payment plugin is tokenising
    token: "tok_stripe_xxx"            # token from that plugin's JS SDK
  ) {
    paymentMethod { id kind last4 }
    errors
  }
}
```

All three arguments are required. `pluginGuid` tells Bunny which
configured Plugin instance the token belongs to — query
`paymentPlugins(accountId:)` to discover the guids available on a
given account. `token` format is plugin-specific: see the tokenisation
SDK of whichever plugin (Stripe, Adyen, etc.) is configured. Bunny
does not accept raw card numbers.

## Consolidation billing

When an account has multiple active subscriptions, you can roll their
invoices into a single consolidated invoice at month-end instead of
sending N separate ones:

```graphql
mutation MergeInvoices {
  invoiceMerge(invoiceIds: ["inv_a", "inv_b", "inv_c"]) {
    invoice { id number amount }        # the new consolidated invoice
    errors
  }
}
```

The originating invoices move to `MERGED` state; the consolidated
invoice references them. `invoiceMergeUndo(invoiceId)` restores the
originals (non-destructive).

Alternatively, set the account's `consolidatedBilling: true` and let
the cron queue invoices into `QUEUED` state automatically; a scheduled
consolidation run finalises them into a `CONSOLIDATED` invoice on a
configurable cadence.

## Tax

Tax is calculated on `PREPARING → READY` via the warren's configured
tax plugin (Avalara, Stripe Tax, etc.). If the plugin times out or
returns an error, the invoice lands in `FAILED`:

```graphql
# After the tax plugin is fixed (e.g. credentials rotated) — valid only
# on invoices currently in FAILED state; other states error with
# "Invoice is not in a failed state".
mutation RetryTax { invoiceRetryTax(id: "inv_xxx") { invoice { id state } errors } }

# To force a recalc after the catalog's taxCode changed — valid on
# non-terminal invoices (not VOIDED, PAID, MERGED, CONSOLIDATED).
mutation UpdateTax { invoiceUpdateTaxes(id: "inv_xxx") { invoice { id state } errors } }
```

The `taxCode` on `PriceListCharge` (see `bunny-catalog`) is the
integrator-facing input; changing it mid-period won't retroactively
fix old invoices without an explicit `invoiceUpdateTaxes`.

## Invoice templates

To customize invoice PDFs (logo, colours, localized text), configure
an `InvoiceTemplate` and attach it via the account or warren defaults:

```graphql
mutation CreateTemplate {
  invoiceTemplateCreate(attributes: {
    name: "Acme — EU invoices"
    header: "Acme Corp, Dublin"
    footer: "VAT: IE1234567"
    logoUrl: "https://cdn.example.com/logo.png"
  }) {
    invoiceTemplate { id name }
    errors
  }
}
```

Templates are a rendering layer — they don't change billing logic.
The full `InvoiceTemplateAttributes` input has ~15 knobs (payment
terms text, signature block, etc.); see the schema.

## Reconciliation patterns (external accounting)

To keep Bunny in sync with QuickBooks / NetSuite / Xero:

1. **Pull incrementally by `updatedAt`.** Query `invoices(filter:
   "updated_at > '<last_sync>'")` and page via Relay cursors (see
   `bunny-graphql`).
2. **Map on `code` or `number`**, not Bunny IDs — codes / invoice
   numbers survive environment promotion; IDs don't.
3. **Mirror state changes, don't try to write back.** If your
   accounting system says an invoice is paid but Bunny says UNPAID,
   Bunny's state machine is the authority. Post a Payment in Bunny
   instead of editing the invoice directly.
4. **Handle VOIDED + MERGED specially** — your accounting system
   may not understand these states. The typical mapping: VOIDED →
   "deleted in accounting"; MERGED → "line-item moved to the target
   invoice"; CONSOLIDATED → "top-level invoice is the one you book."
5. **Drive deltas from webhooks**, not polling. Workflow webhooks
   fire on Invoice / Payment / CreditNote state changes — see
   `bunny-webhooks`.
